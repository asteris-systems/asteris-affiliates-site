# Migrate from WP Overnight PDF Invoices

A walkthrough for moving from WooCommerce PDF Invoices & Packing Slips (WP Overnight / UpdraftPlus, free + Premium add-ons) to the Asteris Affiliates PDF Invoices module. WP Overnight's per-order invoice numbers are copied across automatically by the built-in migration adapter; settings and templates are rebuilt manually.

## TL;DR

- The Asteris **PDF Invoices** module ships with a built-in WP Overnight adapter (`WP_Overnight_PDF_Adapter`) that copies `_wcpdf_invoice_number` to `_asteris_invoice_number` on every existing order, then advances Asteris's counter to `MAX(source) + 1` so newly-issued numbers never collide.
- Plugin settings (`wpo_wcpdf_general_settings`, `wpo_wcpdf_documents_settings_invoice`, `wpo_wcpdf_documents_settings_packing-slip`, `wpo_wcpdf_template_settings`) are **not** copied â€” they are rebuilt on the Asteris settings screen. Most stores finish this in 10-15 minutes.
- Custom theme template overrides at `wp-content/themes/{theme}/woocommerce/pdf/Simple/{invoice,packing-slip}.php` are **not** ported. Asteris uses a different template engine.
- The migration is **non-destructive**: WP Overnight's meta and options are left in place. Rollback is reactivating WP Overnight and disabling the Asteris module.
- Historical PDF files already generated by WP Overnight (if stored on disk) are not regenerated. Re-downloads from Asteris produce a fresh PDF using the Asteris template against the live order data.

## Before you start

1. Take a full database backup. The adapter writes new meta, never edits source meta â€” but a backup is non-negotiable.
2. Stage the migration if possible. Clone the site to a staging URL, run through every step, then repeat on production once you trust the result.
3. Note your current versions: WordPress, WooCommerce, PHP, and the exact WP Overnight build (free core + which Premium add-ons â€” Pro, Templates, etc.). Asteris's PDF Invoices module requires PHP 8.1+ and WooCommerce 8.0+ (matches plugin-wide minimums).
4. Record your **current next invoice number** from WP Overnight: WooCommerce â†’ PDF Invoices â†’ Documents â†’ Invoice â†’ "Next invoice number". You will sanity-check this against Asteris's post-migration value.
5. Export your WP Overnight settings (the plugin has a "Status" / settings export panel) and save a copy of any custom template files from `wp-content/themes/{your-theme}/woocommerce/pdf/`.
6. Deactivate aggressive page caches and any object cache that touches `wp_options` for the migration window. The Asteris adapter advances the `asteris_pdf_invoices_next_number` counter on the final batch â€” a stale cache can mask that.
7. Pick a low-traffic window. The migration itself is fast (well under a minute for stores with thousands of orders) but you do not want a customer placing an order mid-cutover.

## What gets imported

| From WP Overnight | To Asteris | Status |
|---|---|---|
| `_wcpdf_invoice_number` (per-order meta) | `_asteris_invoice_number` (per-order meta) | auto |
| `_wcpdf_invoice_number_data` / `_wcpdf_invoice_number_formatted` (formatted display number with prefix/suffix) | Re-rendered at display time from `_asteris_invoice_number` + Asteris's `invoice_prefix` and `invoice_padding` settings | manual (rebuild prefix to match) |
| `_wcpdf_invoice_date` (ISO issue date) | Not stored â€” Asteris uses the WooCommerce order's `date_created` / `date_paid` for the invoice date | not migrated |
| `_wcpdf_invoice_date_formatted` | Re-rendered from the order date using Asteris's `date_format` setting | manual |
| `_wcpdf_packing-slip_date` / `_wcpdf_packing-slip_date_formatted` | Not stored â€” packing slips render using the live order date | not migrated |
| `wpo_wcpdf_next_invoice_number` (counter option) | `asteris_pdf_invoices_next_number` â€” set by the adapter to `MAX(_wcpdf_invoice_number) + 1` on the final batch | auto |
| `wpo_wcpdf_general_settings` (shop name, address, VAT/ABN, logo, footer, etc.) | Asteris settings: `company_name`, `company_address`, `company_tax_id`, `company_email`, `company_phone`, `company_website`, `logo_id`, `footer_text` | manual (re-enter on the Asteris settings screen) |
| `wpo_wcpdf_documents_settings_invoice` (number format, prefix, suffix, padding, attach-to-email rules) | Asteris settings: `invoice_prefix`, `invoice_padding`, `invoice_start_number`, `attach_completed`, `attach_processing`, `attach_invoice_mail`, `attach_refunded`, `doc_title_invoice` | manual |
| `wpo_wcpdf_documents_settings_packing-slip` (title, fields shown) | Asteris settings: `doc_title_packing`, `show_packing`, `packing_show_notes` | manual |
| `wpo_wcpdf_template_settings` (paper size, font, colours) | Asteris template choice (`template` â€” one of `modern`, `classic`, `minimal`, `branded`, `compact`) + `accent_color` | manual (closest visual match) |
| Theme template overrides at `wp-content/themes/*/woocommerce/pdf/Simple/{invoice,packing-slip}.php` | Asteris theme overrides at `wp-content/themes/{your-theme}/asteris/pdf-invoices/{template}.php` | not migrated (different engine â€” rebuild from scratch) |
| Premium add-on: Proforma invoices, Credit notes | Not supported in Asteris v1.0 | not migrated |
| Premium add-on: Bulk export / FTP / cloud sync | Not supported in Asteris v1.0 â€” Asteris has bulk **print** (combine N orders into one PDF) but not scheduled export | not migrated |
| Premium add-on: Custom document numbering streams per document type | Asteris has a single invoice number stream. Per-order override is supported via the metabox | partial |

## What does NOT get imported

- **Invoice issue dates** (`_wcpdf_invoice_date`). Asteris uses the WooCommerce order date as the invoice date. For historical orders the displayed invoice date will shift from "the day WP Overnight first generated the PDF" to "the day the order was placed/paid". For most stores these are the same day; for stores that generate invoices manually weeks after the order, they will differ.
- **Theme template overrides**. WP Overnight's templates live at `wp-content/themes/{theme}/woocommerce/pdf/Simple/*.php` and use WP Overnight's `WPO_WCPDF_Document` class and helper functions (`$this->get_shipping_address()` etc.). Asteris templates live at `wp-content/themes/{theme}/asteris/pdf-invoices/*.php` and use a different context array (`$company`, `$inv_no`, `$order`, `$accent`). They are not source-compatible.
- **Premium add-on documents**: Proforma invoices, Credit notes, Receipts as separate document types. Asteris has invoice + packing slip in v1.0.
- **Per-document email attachment rules from Premium**. WP Overnight Premium lets you attach different document types to different emails. Asteris attaches invoices only, with per-email-type on/off (`attach_completed`, `attach_processing`, `attach_invoice_mail`, `attach_refunded`).
- **WPML / Polylang per-language template variants**. Asteris v1.0 ships a single template per design â€” translation glue for invoices is on the roadmap, not in v1.0.
- **Cloud storage integrations** (Dropbox / Google Drive auto-upload) shipped by WP Overnight Premium. No Asteris equivalent.
- **Bulk export to ZIP / scheduled email export**. Asteris has bulk **print** (admin selects orders â†’ one combined PDF) but not scheduled batch export.

## Step-by-step

1. Take the backup confirmed in "Before you start". Do not skip.
2. Install **Asteris Affiliates** from `Plugins â†’ Add New â†’ Upload Plugin` (or the WordPress.org repo for the free build). Activate it.
3. Activate your licence at `WooCommerce â†’ Asteris â†’ Licence` if you are on a paid tier. Free is fine for the PDF Invoices module.
4. Enable the **PDF Invoices** module: `WooCommerce â†’ Asteris â†’ Modules` â†’ toggle **PDF Invoices** on. Save.
5. **Leave WP Overnight active.** Do not deactivate it yet. The Asteris adapter detects WP Overnight by checking either `wpo_wcpdf_documents_settings_invoice` / `wpo_wcpdf_general_settings`, or any order with `_wcpdf_invoice_number` (HPOS-safe â€” it checks both `wp_postmeta` and `wp_wc_orders_meta`).
6. Go to `WooCommerce â†’ Asteris â†’ Migrate`. The migration manager scans for known sources and lists **WP Overnight PDF Invoices** with a count of orders detected and a 3-order preview (e.g. `Order #1247 â†’ invoice 1198`). Confirm the preview matches your WP Overnight data.
7. Click **Run migration** for WP Overnight PDF Invoices. The adapter runs in batches via `migrate_batch()`. For each order: it reads `_wcpdf_invoice_number`, writes `_asteris_invoice_number` (only if Asteris does not already have a value â€” re-runs are idempotent), and marks the order as migrated. On the final batch it advances `asteris_pdf_invoices_next_number` to `MAX(_wcpdf_invoice_number) + 1`.
8. Go to `WooCommerce â†’ Asteris â†’ PDF Invoices` and rebuild settings to match your old WP Overnight setup:
   - **Company details**: name, address, tax ID (ABN/VAT), email, phone, website.
   - **Branding**: upload logo (Asteris embeds the logo as a base64 data URI to avoid Dompdf/path issues), pick an accent colour.
   - **Numbering**: set `invoice_prefix` (e.g. `INV-`) and `invoice_padding` to match the format your accountant has been seeing. Leave `invoice_start_number` alone â€” the adapter has already set the counter.
   - **Document titles**: `doc_title_invoice` defaults to `Tax Invoice` (correct for AU/NZ GST-registered businesses); use `VAT Invoice` for EU.
   - **Email attachments**: tick the same emails WP Overnight was attaching to (`attach_completed` is the most common â€” equivalent to WP Overnight's default).
   - **Template**: pick one of `modern`, `classic`, `minimal`, `branded`, `compact`. Closest to WP Overnight's default "Simple" template is **classic**.
9. Open one recent order and click **View Invoice** on the order edit screen. Confirm the invoice number matches what WP Overnight produced. Save it to PDF and check the layout.
10. Run the verification checks in the next section.
11. Once verified, deactivate WP Overnight: `Plugins â†’ Installed Plugins â†’ WooCommerce PDF Invoices & Packing Slips â†’ Deactivate`. Do **not** delete it for at least 30 days â€” keep its data intact in case you need to roll back.
12. After 30 days of clean operation, you may delete WP Overnight. Its meta keys (`_wcpdf_*`) remain in the database as orphan rows â€” harmless, but if you want them gone, run a one-off SQL cleanup (covered in Edge cases below).

## Verifying the migration

1. **Counter sanity check**. Compare `asteris_pdf_invoices_next_number` (visible in `WooCommerce â†’ Asteris â†’ PDF Invoices â†’ Numbering`) against the value you recorded from WP Overnight in step 4 of "Before you start". Asteris's value should equal `MAX(your highest WP Overnight number) + 1`. If WP Overnight said "next: 1247" and your highest issued was 1246, Asteris should show 1247.
2. **Random spot-check**. Pick 5-10 orders across the date range (oldest, middle, newest, plus a couple of recent ones). For each: open the Asteris invoice and confirm the displayed number matches the WP Overnight number (you can still view both side-by-side while WP Overnight is active but disabled at the module level).
3. **Place a test order**. Use a guest checkout or a staff account. Confirm: (a) the new order receives the **next** sequential number (no gap, no collision), (b) the order confirmation email contains the Asteris "Download Invoice" link, (c) if `attach_completed` is on, marking the test order Completed triggers a `.pdf` attachment.
4. **HPOS check** (if you have High-Performance Order Storage enabled). The adapter queries both `wp_postmeta` and `wp_wc_orders_meta`. Confirm that orders created post-HPOS-migration have their `_asteris_invoice_number` in the HPOS meta table.
5. **My Account page**. Log in as a real customer (or impersonate one) and visit `My Account â†’ Orders`. Confirm the **Invoice** action button appears, links to the Asteris URL pattern `/?asteris_invoice={order_key}`, and downloads the correct invoice.

## Rollback

The migration is non-destructive. WP Overnight's data (`_wcpdf_*` meta and `wpo_wcpdf_*` options) is untouched. To roll back:

1. Deactivate the Asteris **PDF Invoices** module: `WooCommerce â†’ Asteris â†’ Modules` â†’ toggle off.
2. Reactivate WP Overnight (if you deactivated it): `Plugins â†’ Installed Plugins â†’ Activate`.
3. Sanity-check one or two orders â€” WP Overnight's "View Invoice" link should work as before, using its original meta.
4. (Optional) Remove the Asteris-only meta written during migration. From a database tool (e.g. WP-CLI, phpMyAdmin), with a backup in hand:
   ```
   DELETE FROM wp_postmeta WHERE meta_key = '_asteris_invoice_number';
   DELETE FROM wp_wc_orders_meta WHERE meta_key = '_asteris_invoice_number';
   DELETE FROM wp_options WHERE option_name = 'asteris_pdf_invoices_next_number';
   ```
   Replace `wp_` with your actual table prefix. The `wp_wc_orders_meta` line is only relevant on HPOS sites.
5. Leave the Asteris plugin installed if you intend to retry later, or uninstall via `Plugins â†’ Delete`. Asteris's uninstaller respects the "keep data on uninstall" toggle.

If rollback is needed because invoice numbers are showing wrong, the most likely cause is a stale object cache â€” flush it (`wp cache flush` or your cache plugin's purge button) before rolling back.

## Edge cases and troubleshooting

### WP Overnight's counter is ahead of the highest issued number

This happens when WP Overnight reserved numbers for invoices that were cancelled or deleted. The Asteris adapter sets `asteris_pdf_invoices_next_number = MAX(_wcpdf_invoice_number) + 1`, which is based on issued numbers, not WP Overnight's reserved counter. If WP Overnight's counter was 1300 but the highest issued number was 1280, Asteris will continue at 1281. To preserve the gap (some accountants want it), manually set Asteris's `Starting number` field to 1301 on the settings screen after migration â€” it will only take effect for the *next* claim, which is what you want.

### HPOS half-migrated stores

If your store is mid-transition between legacy `wp_postmeta` orders and HPOS `wp_wc_orders_meta` orders, some orders will only have `_wcpdf_invoice_number` in postmeta and some only in `wc_orders_meta`. The adapter handles both paths in `is_source_installed()` and `advance_counter()` â€” `MAX()` is taken across both tables. No action needed, but it's worth running the **random spot-check** above with at least one order from each side.

### Custom template overrides in your theme

WP Overnight overrides at `wp-content/themes/{theme}/woocommerce/pdf/Simple/invoice.php` will **not** be picked up by Asteris. Asteris looks at `wp-content/themes/{theme}/asteris/pdf-invoices/{template}.php`. The template context is different â€” Asteris passes a flat array (`$company`, `$inv_no`, `$accent`, `$logo_url`, `$order_date`, `$paid_date`, `$generated_on`, `$footer_text`, `$invoice_note`, `$order` as a real `WC_Order`). You will need to rebuild any custom template against the new context. Copy one of the five bundled templates (e.g. `modern.php`) from the plugin's `src/Modules/PDF_Invoices/Templates/` directory as your starting point.

### Tax invoice numbering must continue from the existing counter, not restart at 1001

This is what the adapter's `advance_counter()` exists for. If after migration you see Asteris starting at 1001 (the default) instead of continuing your sequence, two causes are likely. **(a) The migration didn't finish** â€” `migrate_batch()` only calls `advance_counter()` when the final batch completes (when `next` is `null`). Re-run from `WooCommerce â†’ Asteris â†’ Migrate`. **(b) Object cache is masking the option update** â€” flush cache (`wp cache flush`) and reload the settings screen.

### Per-order invoice number override from WP Overnight Premium

WP Overnight Premium allows manual override of an order's invoice number. The adapter copies whatever value is in `_wcpdf_invoice_number` â€” overridden or auto-issued, it doesn't matter. After migration you can continue to override on a per-order basis using Asteris's **Asteris â€” Invoice** metabox on the order edit screen (sets `_asteris_invoice_number`).

### Proforma / Credit note documents from Premium

Asteris v1.0 does not have separate document types for proforma invoices or credit notes. If your accounting flow depends on these, do not deactivate WP Overnight Premium â€” run both plugins side-by-side: Asteris for the standard tax invoice + packing slip, WP Overnight for proforma + credit notes. There is no conflict (different meta keys, different option groups, different URL endpoints).

## After the migration

1. **Watch the first week's orders.** Confirm every new order receives a unique, sequential invoice number with no duplicates and no gaps. Asteris's number-claim is atomic at the SQL level (single `UPDATE wp_options SET option_value = option_value + 1`) so race conditions are not a real risk, but the first week is when configuration mistakes surface.
2. **Update your accountant.** Send them a sample of the new invoice PDF. Asteris's `Tax Invoice` title, ABN/tax ID placement, and totals breakdown are GST-compliant for AU and VAT-compliant for EU/UK, but every accountant has preferences â€” get sign-off before tax time.
3. **Reconcile one period.** Pick a recent month, pull invoices for that month from Asteris (bulk-print is the fastest way â€” admin orders list â†’ select all â†’ Bulk actions â†’ Print invoices), and compare totals to your accounting software's expected total for that month.
4. **Archive WP Overnight after 30 days clean.** Deactivate is enough for the first 30 days. After that, deletion is safe â€” the `_wcpdf_*` meta remains in the database but is harmless. If you want to clean up, run the `DELETE FROM wp_postmeta WHERE meta_key LIKE '_wcpdf_%'` cleanup (with a backup, and after confirming no other plugin reads those keys).
5. **Document the cutover date** in a private note. If a customer ever asks "why did my invoice number jump format", you'll need to point at the cutover date.
